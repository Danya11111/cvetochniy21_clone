# Платежный контур

## Критичный путь

1. `POST /api/checkout`
   - валидация входных данных;
   - создание/обновление `orders`;
   - вызов T-Bank init;
   - возврат `paymentUrl`.

2. `POST /api/tbank/notify`
   - проверка подписи/валидности уведомления;
   - обновление `payments`;
   - обновление `orders` по статусу;
   - обработка `AUTHORIZED` / `CONFIRMED` / `CANCELED|REJECTED`.

3. Side effects при `CONFIRMED`
   - post-paid DM пользователю;
   - интеграционные действия с МойСклад;
   - event-публикация в operational контур.

## Идемпотентность

- Повторный webhook `CONFIRMED` не должен дублировать критичные side-effects.
- Сохраняются активные защитные флаги в `orders` для недопущения дублей user-facing side-effects.
- Платежный статус в БД остаётся источником истины.

## Что проверять при регрессии

1. Checkout как контракт (`paymentUrl` и корректный order state).
2. Обработку webhook по каждому ключевому статусу.
3. Отсутствие дублей при повторных уведомлениях.
4. Что ошибки новых контуров (topics/outbox/admin) не ломают фиксацию оплаты.

## Текущий вердикт (pre-production)

По результатам актуального regression-pass:
- критичный payment flow сохранён;
- явной регрессии в маршрутах checkout/notify/confirmed не выявлено;
- требуется обязательный ручной e2e smoke перед релизом.
