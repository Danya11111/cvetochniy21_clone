# Action Center (mobile)

## Зачем нужен экран

`Центр действий` — это executive слой, который собирает сигналы из всех ключевых модулей и показывает, что делать сегодня в первую очередь:
- где теряются деньги;
- где риск потери клиентов;
- где быстрые точки роста;
- куда перейти в 1-2 тапа.

## Структура экрана

Файл: `frontend/admin/app.js`

Секции:
1. Executive brief (короткая сводка дня).
2. Top priority (1-3 действия с максимальным приоритетом).
3. Группы действий:
   - Деньги
   - Клиенты
   - Рассылки
   - Поддержка
   - Рост
   - Операции
4. Quick wins (быстрые действия с легким входом).

На `Главной` добавлен компактный блок `Главные действия сегодня` (top 2-3 действия + переход в полный центр).

## Backend action engine

Добавлен endpoint:
- `GET /api/admin/actions/summary`

Реализация:
- `backend/admin-repository.js` -> `getActionsSummary()`
- `backend/admin-routes.js` -> `/actions/summary`

Engine использует агрегаты из:
- `orders summary`
- `clients summary`
- `broadcasts summary`
- `support summary`
- `analytics summary (7d)`

## Формат действия

Каждый action содержит:
- `id`
- `category`
- `priority` (`critical|high|medium|low`)
- `title`
- `message`
- `business_impact_label`
- `impact_value` (если есть корректная оценка)
- `cta_label`
- `target_screen`
- `target_filters`
- `source_signal`
- `freshness`
- `expires_at` (опционально)

## Реализованные правила (core set)

Основные rules-based правила:
1. Замороженная выручка в неоплаченных заказах.
2. Крупные чеки без оплаты.
3. VIP-спящие клиенты.
4. Новые клиенты без повтора.
5. Клиенты-кандидаты на возврат.
6. Диалоги поддержки в ожидании.
7. Критичные обращения поддержки.
8. VIP в ожидании поддержки.
9. Новые клиенты без реакции поддержки.
10. Проблемные рассылки.
11. Потерянный охват в рассылках.
12. Repeatable-кампании (можно повторить).
13. Просадка среднего чека.
14. Рост ожидания поддержки к прошлому периоду.
15. Рост повторных продаж (сигнал для масштабирования).
16. Просадка выручки к прошлому периоду.

## Executive brief generator

Brief формируется rules-based:
- выбирает 2-4 главных сигнала;
- приоритетно включает деньги, VIP-risk, сервисное ожидание и growth-возможности;
- если критичных сигналов нет, показывает calm-state фокус на рост.

## Quick wins

Quick wins — легкие действия с быстрым эффектом:
- repeatable рассылки;
- спящие клиенты;
- новые без повтора;
- крупные чеки;
- поддержка без ответа.

Каждый quick win имеет короткий текст и прямой переход.

## Связь с другими экранами

Action targets ведут в существующие экраны с filter-state:
- `orders`: `unpaid`, `large`, `problematic`
- `clients`: `sleeping`, `vip`, `new`, `repeat`, `return`
- `broadcasts`: `problematic`, `repeatable`, `failed`
- `support`: `waiting`, `critical`, `vip`, `new`
- `analytics`: периодический срез

## Ограничения текущей rules-based модели

- Нет ML/AI приоритизации.
- Нет персонализации по ролям операторов.
- Нет оркестрации автоматических playbooks.

Текущая цель — дать owner/manager четкий ежедневный action layer без фейковой "умности".
