# Миграция данных с RU-сервера на новый сервер («Цветочный 21»)

Документ основан на **аудите кода репозитория** (`backend/db.js`, `backend/config.js`, backend-сервисы, `frontend/app.js`).  
**Не подключается к продакшену**, не содержит учётных данных.

---

## A. Цель миграции

Перенести **все операционные данные** приложения (клиенты, заказы, платежи, админ-доступ, рассылки, поддержку, промо-материалы и т.д.) на новый сервер так, чтобы:

- не потерять ни одну запись SQLite;
- не забыть **файлы вне БД**, на которые есть ссылки в БД (`backend/data/promotion-uploads/`);
- после переключения DNS сохранить работоспособность **checkout**, **Т-Банка**, **МойСклад**, **Telegram webhook**, админки.

---

## B. Что именно переносим

| Область | Где хранится | Комментарий |
|--------|--------------|-------------|
| **Клиенты (профиль бота)** | SQLite: `users` (+ связанное) | Telegram id, имена, бонусы, темы, источники `first_source_code` / `last_source_code`, `first_seen_at` и др. |
| **Адреса** | SQLite: `addresses` | Если используются в продукте. |
| **Заказы** | SQLite: `orders` | **Состав заказа хранится в `items_json`** (отдельной таблицы `order_items` **нет**). |
| **Платежи** | SQLite: `payments` | T-Bank: `payment_id`, сумма в копейках, статус, `raw_json`. |
| **Оплаченность / бонусы заказа** | Колонки в `orders` | `total_paid`, `status`, бонусные поля, `checkout_hash`, `ms_sync_hash`. |
| **МойСклад по заказу** | `orders` | `ms_id`, `ms_name`, `ms_state_name`, **`moysklad_sync_status`**, **`moysklad_sync_error`**. |
| **Каталог (кеш/снимок)** | SQLite: `products` | Если каталог синхронизировался с МойСклад. |
| **Корзина «до оформления» (снимок)** | **Клиент** + SQLite: **`abandoned_carts`** | В браузере: **`f21_cart_blob_v1`** (состав) и стабильный **`f21_cart_key_v1`** (UUID), синхронизируемый на сервер. Снимок для админки; **не** источник истины для сумм при оплате. |
| **Брошенные корзины (abandoned)** | SQLite: **`abandoned_carts`** | Жизненный цикл и уведомления — `backend/abandoned-cart-service.js`, тема форума **`TELEGRAM_TOPIC_ABANDONED_CARTS_ID`**. Переносите записи вместе с БД. |
| **Администраторы Mini App** | SQLite: **`admin_users`** | После bootstrap — источник истины для доступа по WebApp initData; владелец из env. |
| **Аудит админских действий** | `admin_action_logs` | История действий. |
| **Темы форума (реестр)** | `telegram_topics` | Привязка пользователь → топик в супергруппе. |
| **Outbox событий** | `event_outbox` | Надёжная доставка событий. |
| **Рассылки** | `broadcast_campaigns`, `broadcast_deliveries`, `broadcast_trigger_audit`, `broadcast_campaign_events` | Полный контур кампаний. |
| **Поддержка** | `support_threads`, `support_messages`, `support_response_windows` | Треды и сообщения relay. |
| **Флаги runtime** | `runtime_flags` | Оперативные переключатели через админку. |
| **Продвижение (sources / broadcasts)** | `promotion_*` таблицы | Источники, клики, карточки, ключевые слова, галерея. |
| **Идемпотентность webhook Telegram** | `telegram_processed_updates` | Защита от дублей `update_id`. |
| **Загруженные изображения промо-карточек** | **ФС:** `backend/data/promotion-uploads/` | Пути в БД (`image_storage_path`, `promotion_broadcast_images.storage_path`). **Обязательно копировать вместе с БД.** |
| **Статический контент приложения** | Репозиторий / сборка nginx | Образ фронта, статика — из деплоя; **не** из SQLite. |
| **PDF согласия / welcome-картинки** | По `config`: пути к файлам в корне репозитория или URL | Если на старом сервере положены **локальные** файлы (не только в Git) — перенести их вручную по путям из env. |

**Не переносится миграцией БД:** cookies/localStorage состояние «кэша» корзины у клиента (TTL **blob**). Стабильный **`f21_cart_key_v1`** генерируется на устройстве; при первом визите после переноса сервер создаст новые строки **`abandoned_carts`** автоматически. Для непрерывности аналитики **перенесите таблицу `abandoned_carts`**, если она уже накоплена на старом сервере.

## C. Текущая архитектура хранения (фактически по коду)

- **СУБД:** **SQLite 3** (пакет `sqlite3` в Node).
- **Путь к файлу БД:**
  - По умолчанию: **`backend/database.sqlite`** (относительно **рабочего каталога процесса Node** при старте `node backend/server.js`).
  - Или переменная окружения **`F21_SQLITE_PATH`** (абсолютный путь к файлу `.sqlite`/`.db` на проде типично `/var/lib/...`).
- **Миграции схемы:** инициализация и изменения выполняются **приложением** из `backend/db.js`: `CREATE TABLE IF NOT EXISTS`, функция **`ensureColumn`**, дополнительные async-миграции (`runAllMigrationsAsync`). Отдельной папки SQL-миграций в репозитории нет.
- **Персистентность вне SQLite:** каталог **`backend/data/promotion-uploads/`** — бинарные файлы карточек «Продвижение». В `.gitignore` уже есть исключение для этих uploads.
- **Режим журнала WAL:** явного `PRAGMA journal_mode=WAL` в коде не задано; на проде возможны файлы **`*.sqlite-wal`** / **`*.sqlite-shm`** если режим включали вручную — см. секцию резервных копий.
- **Корзины в браузере:** `frontend/app.js` — **`f21_cart_blob_v1`** (TTL по умолчанию 14 дней) и **`f21_cart_key_v1`**. Снимок для отчётности пишется в **`abandoned_carts`** через публичные API; оформление заказа по-прежнему валидирует цены на сервере.

## D. Рекомендованный способ переноса (SQLite)

Подходит для этого проекта.

1. **Согласованная копия БД при остановленной записи**  
   - Остановите процесс Node **или** переключите трафик (maintenance), чтобы не было записи заказов/платежей во время копирования.
2. **Предпочтительно: логический бэкап через SQLite backup API**  
   - Команда: `sqlite3 /path/current.db ".backup '/path/backups/foo.sqlite'"`  
   - Так вы получаете консистентный снимок даже если рядом есть `-wal` (при активном приложении результат всё же менее предсказуем — см. ниже).
3. **Целостность**  
   - `sqlite3 backup.sqlite "PRAGMA integrity_check;"` → ожидается **`ok`**.
4. **WAL/SHM**  
   - Если рядом с БД есть `*.sqlite-wal` и `*.sqlite-shm`, при **остановленном приложении** после checkpoint обычно достаточно одного основного файла после `.backup`. Если копируете «вручную» файлы без backup — перенесите все три файла одновременно **и** приложение должно быть остановлено.
5. **Восстановление на новом сервере**  
   - Положите файл резервной БД как целевой `F21_SQLITE_PATH` (или замените `backend/database.sqlite` согласно вашему `systemd` и cwd).  
   - Запустите приложение один раз миграции применятся к существующей схеме автоматически (`ensureColumn`).
6. **Файлы промо-картинок**  
   - Синхронно скопируйте **`backend/data/promotion-uploads/`** (или ту же диреструктуру на новой машине под тем же относительным путём к `promotion-service.js`).

**PostgreSQL/MySQL:** в этом репозитории основной режим работы приложения не использует их; отдельного дампа не требуется, если только вы не ведёте **параллельную** внешнюю БД (в коде не обнаружено).

## E. Zero / minimal downtime plan

1. Развернуть новый сервер (ОС, Node ≥18, nginx, systemd, firewall).
2. Склонировать нужный релиз, `npm ci` / установка зависимостей по вашему runbook (`deploy/`).
3. Поднять **`/etc/...`** env файл **без** продакшен-секретов в Git; указать **`F21_SQLITE_PATH`**, **`APP_PUBLIC_URL`**, ключи Telegram, T-Bank, МойСклад.
4. На **старом** сервере: объявить maintenance **или** остановить systemd-юнит приложения перед финальным снимком.
5. Финальный **backup SQLite** (+ копирование `promotion-uploads`).
6. Перенести архив по **scp/rsync** на новый сервер (отдельный канал, права доступа только у админов).
7. На новом: **restore** (замена файла БД + синхронизация каталога uploads).
8. `PRAGMA integrity_check` и **counts** по таблицам (скрипт `scripts/migration/validate.sh`).
9. Запустить приложение на новом; smoke: `/api/health` / админ-хелсы по вашему runbook.
10. Smoke: авторизация в админке, список заказов, открыть заказ с `items_json`.
11. **Не переключаем DNS**, пока не подтверждены платежи/webhook на **новом** публичном URL (staging-домен или временный HTTPS).
12. Telegram: после финального домена — `setWebhook` с тем же секретом/URL.
13. Т-Банк: **`APP_PUBLIC_URL`** и **NotificationURL** должны указывать на финальный домен (см. `backend/tbank.js`).
14. Переключить DNS A/AAAA или reverse proxy upstream на новый сервер.
15. Мониторить логи 24–48 часов.

## F. Rollback plan

Если после переключения выявилась ошибка данных или интеграций:

1. **Вернуть DNS / upstream на старый сервер**, если он ещё **read-only безопасен** или кратко **включить запись только на одном** месте (никогда два writable одновременно).
2. **Не восстанавливать** случайный дамп «поверх живой базы без копии» — см. секцию запретов.
3. Сохранить дамп **нового** сервера на время разбора.
4. **Split-brain риск:** если на новом уже принимали заказы или webhook оплат — откат DNS **не восстановит автоматически** эти заказы на старой БД. Тогда нужен **manual merge / реимпорт** — этого избегают, держа старый сервер только read-only перед финальным cutover или продлив maintenance.

## G. Validation checklist

### БД (до и после переноса одинаковая логика)

- [ ] `PRAGMA integrity_check` → `ok`
- [ ] `COUNT(*)` пользователей: `users`
- [ ] `COUNT(*)` заказов: `orders`
- [ ] **«order_items»:** отдельной таблицы нет — достаточно выборки нескольких последних `orders.items_json` для глаз
- [ ] Платежи: `payments`
- [ ] Админы: `admin_users`; запись с **`telegram_id`** владельца (`ADMIN_OWNER_TG_ID`, например **67460775**) и флаг **`is_owner=1`** (если добавляется при bootstrap — проверить)
- [ ] Аудит (опционально): `admin_action_logs`
- [ ] Последние 10 заказов: сравнить `id`, `created_at`, `status`, `total_paid`
- [ ] Последние изменения профилей: по `telegram_id` / `updated` полям где есть (`users` имеет добавленные поля по мере версий — ориентир `first_seen_at`, заказы новых клиентов)
- [ ] Telegram: `telegram_topics`
- [ ] Рассылки: хотя бы `broadcast_campaigns` count
- [ ] Поддержка: `support_threads`, `support_messages`
- [ ] Promo: `promotion_broadcasts`, `promotion_broadcast_images`, и **фактическое наличие файлов** по `storage_path` в `backend/data/promotion-uploads/`
- [ ] Outbox: `event_outbox` (остаток `NEW` не обязан совпадать до сравнения моментов времени снимка)
- [ ] Серверные брошенные корзины: `COUNT(*)` из `abandoned_carts`; при необходимости — разбивка по `status` (см. вывод `scripts/migration/validate.sh`)
- [ ] Не ожидать «переноса» localStorage с устройств клиентов для аналитики брошенных корзин — переносится **таблица** `abandoned_carts` и файлы из § B

### Функциональные проверки

- [ ] Открывается админка с initData
- [ ] Создаётся тестовый заказ / доходит до `PENDING_PAYMENT` / нет «сырых» ошибок на фронте
- [ ] МойСклад при ошибке **не ломает** checkout (уже заложено в логике сервера — повторить на новом URL)
- [ ] Тестовый callback Т-Банка (или Init) с нового домена
- [ ] Нет **захардкоженного** старого домена в env (проверка `APP_PUBLIC_URL` / `BASE_URL`)

## H. Команды миграции (шпаргалка)

См. готовые скрипты в **`scripts/migration/`** и их **`README.md`**.

Минимум вручную:

```bash
# Бэкап (при остановленном приложении предпочтительно)
sqlite3 "/var/lib/cvetochny21/app.sqlite" ".backup '/safe/path/f21-$(date -u +%Y%m%dT%H%M%SZ).sqlite'"

# Проверка
sqlite3 "/safe/path/f21-....sqlite" "PRAGMA integrity_check;"

# Пример counts
sqlite3 "/safe/path/f21-....sqlite" "SELECT 'users', COUNT(*) FROM users UNION ALL SELECT 'orders', COUNT(*) FROM orders;"
```

Перенос файлов промо:

```bash
rsync -aH --progress ./backend/data/promotion-uploads/ user@new:/opt/cvetochny21/backend/data/promotion-uploads/
```

## I. Что нельзя делать

- Нельзя переносить **только код** без SQLite + uploads, если нужны исторические заказы и клиенты.
- Нельзя держать **две writable** копии одной и той же логической БД (два принимающих заказа инстанса с одной и той же «истиной» без репликации).
- Нельзя **коммитить** в Git: дампы БД, `.env`, архивы с PII, `backups/*.sqlite`.
- Нельзя делать **restore поверх** рабочей БД **без** предварительной копии текущего файла.
- Нельзя менять **боевой DNS** до smoke-тестов на новом URL.
- Нельзя забыть **Telegram webhook** и **публичные URL Т-Банка** после смены домена.
- Нельзя считать перенесёнными **«корзины»** клиентов в браузере — они живут только на устройствах.

## Риски потери данных (кратко)

| Риск | Митигация |
|------|-----------|
| Корзина до checkout в localStorage | Не восстанавливается из БД; пользователи теряют несохранённые корзины при смене устройства/очистке — это уже так в продукте. |
| Abandoned carts не в БД | Нечего верифицировать в SQL; следующая версия должна добавить таблицы + повторить миграционный контроль. |
| Ордер во время downtime | Принимают maintenance и повтор заказов; минимизация времени окна cutover. |
| Забытые файлы промо-uploads | Сверка `promotion_broadcast_images` и листинг каталога на диске. |
| Split-brain между старым и новым после переключения | Одна активная база записи «истины» или явный режим только чтения на старом. |
